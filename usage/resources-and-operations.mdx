The AzuraCast node uses a two-level selector to navigate the AzuraCast API:

1. **Resource** — the API domain or tag (e.g. `Stations Media`, `Administration Users`)
2. **Operation** — the specific endpoint within that resource (e.g. `List Files`, `Upload File`)

Selecting a resource filters the operation list to only show endpoints that belong to that tag. Selecting an operation then shows only the input fields that endpoint requires.

## How operations are named

Each operation's display name is derived from the AzuraCast OpenAPI specification using this priority order:

1. **Operation ID** — if present, the `operationId` string is converted to title case (e.g. `getStationMedia` → `Get Station Media`)
2. **Summary** — if the operation ID is absent or path-like, the `summary` field is used
3. **Description** — used as a fallback when neither of the above produces a clean label
4. **HTTP method fallback** — if none of the above are usable, the label becomes `GET request`, `POST request`, etc.

Redundant resource tokens are stripped from operation names to keep them concise within context. For example, an operation named `Get Station Playlist` inside the **Stations Playlists** resource becomes simply **Get Playlist**.

When multiple operations in the same resource would produce identical display names, the HTTP method is appended (e.g. `Update (PUT)`, `Update (PATCH)`) and further numbered if necessary.

## Field visibility

Each operation shows only the fields that apply to that specific endpoint:

- **Path parameters** are shown as required string fields (e.g. `Station ID`, `ID`)
- **Required query parameters** are shown as required fields
- **Optional query parameters** are grouped under an **Optional Query Parameters** collection
- **Required body fields** are shown as required fields
- **Optional body fields** are grouped under an **Optional Body Fields** or **Optional Multipart Fields** collection

Fields from other operations are hidden, keeping each action's configuration panel clean and focused.
